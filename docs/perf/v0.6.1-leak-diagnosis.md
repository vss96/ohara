# v0.6.1 CoreML memory-leak diagnosis

**Date:** 2026-05-02. **RFC:** `docs/superpowers/specs/2026-05-02-ohara-v0.6.1-coreml-leak-rfc.md`. **Plan:** `docs/superpowers/plans/2026-05-02-ohara-plan-7-coreml-leak.md`.

This document records the Phase 1 findings for the CoreML embedder leak documented in the RFC, and the Path A vs Path B verdict that drives the v0.6.1 release.

## TL;DR

- Reproduction confirmed in a tight `embed_batch` loop on Apple Silicon (macOS 26.2, M-series, 24 GB).
- Sustained leak rate **≈ 4.3 MB/batch virtual** in the no-rebuild path; spikes to ~16 MB/batch during ANE warm-up.
- Leak source is heap-attributable (`MALLOC_LARGE` zone, fastembed/ort symbols). RFC hypotheses (1) and (4) are consistent with the data; hypothesis (3) (pure `CoreML.framework` / ANE-side retention) is not — `IOAccelerator` regions stayed flat.
- `phys_footprint` is **not** the right metric — macOS's compressor squashes inactive pages, so `phys_footprint` plateaus while *virtual* size and the kernel-attributed compressed bytes keep climbing. `phys_footprint` plateau is illusory.
- Rebuild-cadence probe (drop and reconstruct `TextEmbedding` every K batches) **mitigates by ~2×** at K=50 (4.3 → 2.0 MB/batch) but does **not** bound growth. Path A bar — "rebuild-cadence holds peak footprint below an OOM-safe ceiling at QuestDB-scale workloads" — is not met.
- **Verdict: Path B.** Ship v0.6.1 with `--embed-provider auto` resolving to CPU for long index passes on Apple Silicon; explicit `--embed-provider coreml` proceeds with a warning. Re-open the in-tree fix as a v0.6.2 / v0.7 candidate after upstream investigation in fastembed / ort.

## Harness

`tests/perf/coreml_leak_repro.rs`, `#[ignore]`'d. Knobs documented in the file. Stays as a regression guard regardless of which path lands.

```sh
cargo build --release --features ohara-embed/coreml -p ohara-perf-tests --tests
LEAK_PROVIDER=coreml LEAK_BATCHES=300 LEAK_BATCH_SIZE=16 LEAK_TEXT_LEN=800 \
    cargo test --release --features ohara-embed/coreml \
        -p ohara-perf-tests --test coreml_leak_repro -- --include-ignored --nocapture
```

Workload shape: batch size 16, ~800-byte (~200-token) salted strings, BGE-small-en-v1.5 model, CoreML execution provider via fastembed 4.9 → ort.

## Findings

### Phase 1 — baseline (Task 1.1, 1.2, 1.3)

300-batch CoreML run. `vmmap --summary` snapshots at t=15 s (post-load), t=120 s (~150 batches), t=240 s (~290 batches).

| Snapshot       | VIRTUAL | RESIDENT | DIRTY  | SWAPPED |
| -------------- | ------- | -------- | ------ | ------- |
| start          | 7.5 GB  | 3.4 GB   | 3.3 GB | 2.8 GB  |
| mid (t=120 s)  | 8.7 GB  | 1.4 GB   | 1.3 GB | 6.0 GB  |
| late (t=240 s) | 8.8 GB  | 6.1 GB   | 5.9 GB | 1.4 GB  |

- **Virtual size grew +1.3 GB across 300 batches ≈ 4.3 MB/batch.** This is the unbounded axis.
- The `RESIDENT ↔ SWAPPED` flip between mid and late is the macOS compressor working: dirty inactive pages get squashed (counted as `swapped`), then re-faulted when the embedder touches them again. Net memory pressure on the host is roughly `RESIDENT + SWAPPED`, not `RESIDENT` alone.
- `phys_footprint` (sampled via `footprint -p`) plateaued at ~7.5 GB after the first ~50 batches, despite continued virtual growth — confirming that `phys_footprint` undercounts the leak.

The growing region is **`MALLOC_LARGE`**:

| Snapshot | MALLOC_LARGE virtual |
| -------- | -------------------- |
| start    | 5.8 GB               |
| mid      | 6.7 GB               |
| late     | 6.7 GB               |

`IOAccelerator` and `IOAccelerator (graphics)` regions stayed flat at 64 KB / 720 KB across all three snapshots — no runaway ANE-side allocation.

This pins the leak to heap allocations on the inference path (fastembed / ort / ort's CoreML EP), not to `CoreML.framework` retention or ANE program state. RFC hypotheses (1) and (4) are consistent; hypothesis (3) is ruled out for the user-visible leak.

At QuestDB scale (~5 800 commits, ~5–10 batches each ⇒ ~30 000+ `embed_batch` calls), 4.3 MB/batch projects to ~129 GB virtual. In practice the kernel's jetsam threshold trips before that — the original kill happened at 32 GB compressed, consistent with ~6–8 k batches into the run.

### Phase 1 — growth axis (Task 1.4)

The 300-batch run already showed the relevant axes:

- **Per-batch:** sustained ~4.3 MB/batch growth tracks `embed_batch` calls. Yes.
- **Per-wallclock with ANE on:** the 50-batch sub-window grew at ~16 MB/batch (warm-up). The 50→300 batch sub-window grew at ~4 MB/batch. Per-batch dominates after warm-up.
- **Per-session-lifetime:** the rebuild-cadence probe (next section) shows per-session contribution exists but does not dominate.

Verdict: leak is **per-batch-with-CoreML-EP-attached**. Rate-limiting batch dispatch (Task 1.4 fallback) would not help.

### Phase 1 — rebuild-cadence probe (Task 2.1)

Single concrete probe — drop and reconstruct `TextEmbedding` every K batches inside the harness, re-running `apply_provider_to_init` (`crates/ohara-embed/src/fastembed.rs:108-115`) so the new session keeps CoreML attached.

| Run            | Batches | Virtual start | Virtual mid (t=105 s) | Δ virtual | Per-batch leak | Wallclock per batch |
| -------------- | ------- | ------------- | --------------------- | --------- | -------------- | ------------------- |
| K=∞ (control)  | 300     | 7.5 GB        | 8.7 GB → 8.8 GB       | +1.3 GB   | **4.3 MB**     | 1057 ms (warm-up heavy), 790 ms steady |
| K=50           | 200     | 7.5 GB        | 7.9 GB                | +0.4 GB   | **2.0 MB**     | 783 ms                                 |
| K=200          | 200     | 7.5 GB        | (1 rebuild at end ⇒ effectively no-rebuild) | — | — | 809 ms          |

K=50 reduces the leak rate by **~2×** but does not eliminate it. Wallclock per batch is essentially unchanged from the no-rebuild baseline — model rebuild from cache is cheap (~1–2 s amortised over 50 batches).

`SWAPPED` shifts to a different distribution under K=50 (start: 16 KB swap; mid: 16 KB swap, vs 6.0 GB swap at the same point in the no-rebuild run) — the compressor isn't accumulating pressure because the rebuild is releasing pages. This is good news for `phys_footprint`-based metrics, bad news for the underlying leak: per-batch growth is still positive, so the runaway is only delayed.

K=200 was not a useful data point in this 200-batch harness (the rebuild only fires at iter 200, the final iteration). A K=200 sweep would need ≥ 600 batches and was de-scoped per the user's choice of the short sweep.

### Phase 2 — fix-shape decision (Task 2.2)

**Path A bar:** "rebuild-cadence holds peak `phys_footprint` < 4 GB across the full harness, with rebuild overhead < 10% wallclock vs the no-rebuild CPU baseline."

This bar fails on its first clause: the CoreML session's baseline (model + EP init) is ~6.3 GB resident even before any batches run. There is no rebuild K that brings the working set below 4 GB; that bar was written against the RFC's optimistic peak-RSS expectation, which the data refutes.

**Reformulated bar:** "rebuild-cadence flattens per-batch virtual growth to ≤ 0 MB/batch (steady state)." This is the operational test for whether the leak is bounded.

K=50 yields ~2 MB/batch — positive, not flat. To bound growth via rebuild we would need K ≪ 25, at which point session reload overhead dominates wallclock and rank quality (ANE warm-up nondeterminism) becomes a separate problem.

**Verdict: Path B.** Ship v0.6.1 with auto-downgrade for long index passes; keep CoreML viable for short-lived `query` and `index --incremental` calls; reopen the upstream-style fix when fastembed / ort have a release that addresses the heap accumulation.

## Follow-ups (not blocking v0.6.1)

- File an upstream issue against fastembed-rs and/or ort referencing this diagnosis: heap accumulation on repeated `TextEmbedding::embed` calls with the CoreML execution provider attached, ~4 MB/batch sustained.
- Re-run the rebuild-cadence sweep with K=10, K=25 once an upstream fix lands — if the residual is small, K=10 may be acceptable as a belt-and-braces measure.
- Add a `vmmap`-virtual sampler to the harness so future runs don't have to interleave shell snapshots manually.
