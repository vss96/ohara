# ohara v0.5 — `explain_change` MCP tool (capability B)

**Status:** spec.

**Goal:** ship the second of the two v1 MCP tools promised in the
original spec. Where `find_pattern` answers "how was X done before?",
`explain_change` answers "why does THIS code look the way it does?" —
given a file + line range, it returns the commits that introduced and
shaped that code, ordered newest-first, with diff excerpts and
authorship metadata.

## Non-goals

- **Symbol lineage** (track a function across renames / moves).
  Deferred to v0.6 — needs cross-commit symbol matching, which is its
  own design problem.
- **LLM-generated change summaries.** The tool returns the actual
  commit messages + diffs; semantic summarization stays out of scope
  for the MVP. (Augment-style local-LLM summaries are still listed in
  the v0.3 deferred-ideas section.)
- **Cross-rename tracking.** If a file was renamed, blame may not
  reach further back than the rename without `--follow`-style history
  rewriting; v0.5 documents this limitation, doesn't solve it.
- **Time-range or author filtering.** Add via parameters in v0.6+ if
  there's demand.

## Tool surface

### MCP tool: `explain_change`

```jsonc
// input
{
  "file": "src/auth.rs",   // required: repo-relative path
  "line_start": 42,        // optional; default 1
  "line_end": 78,          // optional; default end-of-file
  "k": 5,                  // optional; default 5, clamped 1..=20
  "include_diff": true     // optional; default true
}

// output
{
  "hits": [
    {
      "commit_sha": "abc1234",
      "commit_message": "Switch fetch to retry with backoff",
      "commit_author": "alice@example.com",
      "commit_date": "2025-09-12T14:32:00Z",
      "blame_lines": [42, 43, 44],            // which queried lines this commit owns
      "file_path": "src/auth.rs",
      "diff_excerpt": "+    for _ in 0..3 { ... }",
      "diff_truncated": false,
      "provenance": "EXACT"                    // blame is git-truth, not inferred
    },
    ...
  ],
  "_meta": {
    "lines_queried": [42, 78],
    "commits_unique": 5,
    "blame_coverage": 1.0,                     // 1.0 = every queried line was attributed
    "limitation": null                         // or e.g. "file was renamed; older history not reached"
  }
}
```

### CLI command: `ohara explain <file> [--lines a:b] [--k N]`

Provides a way to test the tool directly without going through MCP.
Output is JSON (matches the MCP body) so it's pipeable into `jq`.

## Architecture

```
explain_change(file, line_start, line_end, k)
    │
    ├─► ohara_git::Blamer::blame_lines(file, line_start..=line_end)
    │       (uses git2::Repository::blame_file)
    │
    ▼
unique commit SHAs → HashSet<String>
    │
    ▼
for each sha (in commit-ts desc order):
    storage.get_commit(repo_id, sha)             ── CommitMeta
    storage.get_hunks_for_file(repo_id, sha, file) ── Vec<Hunk>
    │
    ▼
join blame line attribution back per-commit → ExplainHit
    │
    ▼
truncate to k → response
```

The retrieval layer is **deterministic** — no embeddings, no rerank.
git blame is exact, so `provenance = "EXACT"` for every result.

## Component changes

### `ohara-git` (new module: `blame.rs`)

```rust
pub struct Blamer {
    repo: Arc<Mutex<Repository>>,    // mirror the GitWalker pattern
}

impl Blamer {
    pub fn open(path: &Path) -> Result<Self>;

    /// Blame a contiguous line range. Returns one entry per
    /// distinct commit, with the lines it owns within the range.
    pub fn blame_range(
        &self,
        file_path: &str,
        line_start: u32,    // 1-based
        line_end: u32,      // inclusive, 1-based; clamped to file length
    ) -> Result<Vec<BlameRange>>;
}

pub struct BlameRange {
    pub commit_sha: String,
    pub lines: Vec<u32>,    // sorted 1-based line numbers within this commit's blame
}
```

Implementation uses `git2::Repository::blame_file` with default
options (no `--follow`; v0.5 doesn't track renames). Performance
acceptable on files up to ~10k lines; flag larger files in the
returned `_meta` for future optimization.

### `ohara-core::storage` (Storage trait additions)

```rust
#[async_trait]
pub trait Storage: Send + Sync {
    // ...existing...

    /// Fetch a single commit's metadata. Returns `Ok(None)` if the
    /// SHA isn't indexed (e.g., commit is older than the watermark).
    async fn get_commit(
        &self,
        repo_id: &RepoId,
        sha: &str,
    ) -> Result<Option<CommitMeta>>;

    /// Fetch the hunks of a commit that touch a specific file path.
    async fn get_hunks_for_file_in_commit(
        &self,
        repo_id: &RepoId,
        sha: &str,
        file_path: &str,
    ) -> Result<Vec<Hunk>>;
}
```

Both are simple SQL lookups against existing tables. No new tables.
No migration.

### `ohara-core` — new `explain` module

```rust
// crates/ohara-core/src/explain.rs

pub struct ExplainQuery {
    pub file: String,
    pub line_start: u32,
    pub line_end: u32,
    pub k: u8,
    pub include_diff: bool,
}

pub struct ExplainHit {
    pub commit_sha: String,
    pub commit_message: String,
    pub commit_author: Option<String>,
    pub commit_date: String,
    pub blame_lines: Vec<u32>,
    pub file_path: String,
    pub diff_excerpt: String,
    pub diff_truncated: bool,
    pub provenance: Provenance,    // always EXACT
}

pub struct ExplainMeta {
    pub lines_queried: (u32, u32),
    pub commits_unique: usize,
    pub blame_coverage: f32,        // fraction of queried lines attributed
    pub limitation: Option<String>,
}

pub trait BlameSource: Send + Sync {
    async fn blame_range(
        &self, file: &str, line_start: u32, line_end: u32,
    ) -> Result<Vec<BlameRange>>;
}

pub async fn explain_change(
    storage: &dyn Storage,
    blamer: &dyn BlameSource,
    repo_id: &RepoId,
    query: &ExplainQuery,
) -> Result<(Vec<ExplainHit>, ExplainMeta)>;
```

Pure orchestration; no I/O of its own. Keeps `ohara-core` git-free
via the `BlameSource` trait (mirrors `CommitSource` / `SymbolSource`).

### `ohara-mcp` — new tool `explain_change`

Mirrors the existing `find_pattern` tool's layout
(`tools/explain_change.rs`). Tool description follows the discoverability
pattern from Plan 1 (Layer 1 trigger phrasing — "use when user asks why
this code exists / where this came from / git archaeology"). The same
`ResponseMeta { index_status, hint }` block from `find_pattern` is
appended.

### `ohara-cli` — new `explain` subcommand

```
ohara explain <file> [--lines START:END] [--k N] [--no-diff] [<repo>]
```

Default repo is current dir. JSON output to stdout.

## Edge cases

1. **File not in HEAD.** Return `_meta.limitation = "file does not
   exist in HEAD"`, hits empty.
2. **Line range out of bounds.** Clamp to file length; if
   `line_start > line_end` after clamping, return empty hits with
   `limitation = "line range outside file"`.
3. **Commit not yet indexed.** Blame may report a SHA that's older
   than the watermark. Skip those (with a debug log) and reflect
   in `_meta.commits_unique`. The user can `ohara index` to backfill.
4. **File renamed.** Blame stops at the rename boundary by default.
   If the requested lines are entirely from before a rename,
   `blame_coverage < 1.0` and `_meta.limitation =
   "file was renamed at <sha>; pre-rename history not reached"`.
   v0.5 doesn't follow renames; v0.6 might add `--follow`-style
   support.
5. **Empty repo / no commits.** `blame_range` returns an empty Vec;
   tool returns empty hits + `commits_unique = 0`.

## Out of scope (deferred to v0.6+)

- `--follow`-style rename tracking
- Symbol-mode (`{ symbol: "Foo::bar" }`) — needs lineage.
- LLM-summarized change descriptions.
- Time-range / author filters.
- Multi-file blame in one call.
