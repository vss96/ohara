# ohara v0.4 — Java + Kotlin language support

**Status:** spec.

**Goal:** index Java 17/21+ and Kotlin 1.9/2.0+ source files end-to-end —
symbol extraction, AST sibling-merge chunking, FTS5 BM25 lanes, and
cross-encoder rerank — so `find_pattern` returns useful results on JVM
codebases (with particular emphasis on Spring-flavored idioms where
annotations carry significant semantic load).

## Non-goals

- New MCP tools (no `explain_change` yet — that's v0.5).
- Targeted annotation filtering (`find_pattern { annotation: "@Service" }`).
  Annotations are searchable via embeddings + BM25 because they sit in
  `source_text`, but no dedicated metadata column or query parameter is
  added. v0.5+.
- Java/Kotlin parsers for partial files / broken syntax recovery.
- Lambda or anonymous-inner-class extraction. Symbols stay = named
  callable units (matches the Rust/Python contract).

## Symbol extraction

### Java (target: Java 17+; tree-sitter-java grammar must support sealed
types, records, annotation declarations)

| AST node | `SymbolKind` | Notes |
|----------|--------------|-------|
| `class_declaration` (incl. `sealed`, `non-sealed`, `final`, `abstract`) | `Class` | All modifiers preserved in source_text |
| `interface_declaration` (incl. `sealed`) | `Class` | `permits` clause stays in source_text |
| `enum_declaration` | `Class` | |
| `record_declaration` | `Class` | Java 14+ |
| `annotation_type_declaration` | `Class` | Custom annotation defs |
| `method_declaration` | `Method` | |
| `constructor_declaration` | `Method` | Same kind, different name (the constructor's class name) |

Nested classes / inner classes ARE extracted (one Symbol per nested
class). Lambdas and anonymous inner classes are NOT extracted.

### Kotlin (target: Kotlin 1.9/2.0; grammar must support sealed,
data classes, objects, companions)

| AST node | `SymbolKind` | Notes |
|----------|--------------|-------|
| `class_declaration` (incl. `sealed`, `data`, `inner`, `inline value`) | `Class` | |
| `object_declaration` | `Class` | Kotlin singletons map to Class |
| `companion_object` | `Class` | |
| `function_declaration` (top-level) | `Function` | |
| `function_declaration` (inside class/object/interface) | `Method` | |

Property declarations and Kotlin lambdas are NOT extracted.

## Annotations stay in source_text

The symbol's `span_start` / `span_end` extends to cover preceding
annotations and modifiers. Concretely:

```java
@RestController
@RequestMapping("/users")
public class UserController {  // <-- span_start *here* includes both annotations
    @GetMapping("/{id}")
    public User get(@PathVariable Long id) { ... }
}
```

Result:
- One `Class` symbol named `UserController` whose `source_text` starts
  with `@RestController\n@RequestMapping("/users")\npublic class
  UserController { ... }`.
- One `Method` symbol named `get` whose `source_text` starts with
  `@GetMapping("/{id}")\npublic User get(@PathVariable Long id) { ... }`.

Why: this is the cheapest path to "Spring-aware retrieval" — embedding
models pick up `@RestController` / `@Service` / `@Configuration` as
strong semantic tokens, and FTS5 BM25 on hunk text + symbol name
catches exact matches. No schema change. No new query syntax.

The same convention applies to Kotlin annotations (`@Component`,
`@Configuration`, `@Bean`, `@RequestMapping`, etc.).

## Language detection in `extract_for_path`

File extension dispatch:

| Extension | Module |
|-----------|--------|
| `.rs` | `rust::extract` (existing) |
| `.py` | `python::extract` (existing) |
| `.java` | `java::extract` (new) |
| `.kt`, `.kts` | `kotlin::extract` (new) |

Other extensions return an empty symbol set (current behavior).

## Grammar version pins

`tree-sitter-java`: pin to a recent crate version (≥ 0.21) that supports
sealed types and records. Verify during plan execution; the canonical
crate `tree-sitter-java` on crates.io is actively maintained and tracks
the official tree-sitter org grammar.

`tree-sitter-kotlin`: the canonical grammar lives at
`fwcd/tree-sitter-kotlin` (Rust crate `tree-sitter-kotlin`). It supports
sealed classes, data classes, objects, and companion objects in current
versions. Plan execution validates the chosen pin against fixture
files. If neither the canonical crate nor an actively maintained fork
compiles cleanly against `tree-sitter = "0.22"`, the plan execution
flags this as a blocker and v0.4 ships **Java only** (Kotlin moves to
v0.4.1 or v0.5 depending on what's wrong).

## Out of scope (deferred to v0.5+)

- Annotation filtering as a `find_pattern` parameter
- Symbol-kind metadata for `Annotation`, `Record`, `Object`,
  `Companion` (today they all collapse to `Class`)
- Field/property indexing
- Kotlin extension function detection (currently they'd be `Function`
  via the top-level rule)
- Cross-language linking (Java → Kotlin via JVM interop)
- Imports / package structure as searchable signals

## Files affected

- `Cargo.toml` (workspace) — add `tree-sitter-java` and
  `tree-sitter-kotlin` to `[workspace.dependencies]`
- `crates/ohara-parse/Cargo.toml` — depend on the two new grammars
- `crates/ohara-parse/src/lib.rs` — add `pub mod java; pub mod
  kotlin;`, dispatch from `extract_for_path`
- `crates/ohara-parse/src/java.rs` — new
- `crates/ohara-parse/src/kotlin.rs` — new
- `crates/ohara-parse/src/chunker.rs` — no changes (language-agnostic
  contract holds)

No storage / migration / retriever / MCP changes. v0.4 is purely a
parse-layer addition.
