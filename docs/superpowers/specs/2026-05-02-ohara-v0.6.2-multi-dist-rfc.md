# ohara v0.6.2 — per-host distribution variants RFC

**Status:** RFC — design. Goal is a concrete shape for the v0.6.2 release;
the implementation steps live in the matching plan-8.

**Goal:** make `curl-pipe-sh` install and `ohara update` hand the user
a binary that matches their host's hardware capability — CoreML on
Apple Silicon, CPU everywhere else for now — without forcing a source
rebuild.

## Why this matters

v0.6.1 ships an auto-downgrade for the CoreML memory leak, but that
code path never runs on the released cargo-dist binary. Reason: the
released artifact is built without `--features coreml`, so on Apple
Silicon `EmbedProvider::CoreMl` falls into `apply_provider_to_init`'s
`#[cfg(not(feature = "coreml"))]` arm and returns "embed-provider=coreml
is not enabled in this build". The combination of:

- `detect_provider()` preferring CoreML on Apple Silicon (cfg!), and
- `apply_provider_to_init` rejecting CoreML when the feature is off,

means a freshly-installed v0.6.1 user on an M-series Mac who runs
`ohara index <repo>` with `--embed-provider auto` (the default) hits
that error before the auto-downgrade ever fires for a long pass —
because the auto-downgrade only kicks in for `commits_to_walk >=
LONG_PASS_THRESHOLD`. Short index passes and `query` calls fail
hard, end-to-end, on the released binary.

The README's existing answer is "build from source with `--features
coreml`". That is acceptable for contributors but not for the
"point your MCP client at the binary" install path.

## Non-goals

- Runtime feature detection (dlopen CoreML / CUDA at startup). Big
  effort, cross-cuts ort, deferred.
- One fat binary that links every accelerator. Not portable —
  CoreML.framework only exists on macOS-aarch64; CUDA needs the toolkit
  on a Linux host. The fat-binary shape doesn't solve the cross-platform
  problem, just shifts it.
- CUDA in v0.6.2. CUDA needs a CI runner with the toolkit installed;
  scope-creeps the release. Defer to a follow-up once the CoreML path
  is shipped.

## Shape

Two-pronged:

1. **Default install per platform picks the platform-appropriate
   accelerator.** `aarch64-apple-darwin` ships `--features coreml` from
   v0.6.2 onward; other targets stay CPU-only. Naming in the dist
   manifest stays per-target-triple (no extra `-coreml` suffix on the
   default asset) so axoupdater and the curl-pipe-sh installer keep
   working without changes.
2. **Explicit CPU-only opt-out for users who want the smaller / link-
   stable artifact.** Publish a parallel `ohara-cli-aarch64-apple-darwin-
   cpu.tar.xz` for the audience that wants pure CPU on Apple Silicon
   (CI runners, deterministic benchmarks, hosts where the CoreML
   framework link is broken). Out-of-band — not pulled by `ohara
   update`, only by manual download.

The auto-downgrade we shipped in v0.6.1 then becomes load-bearing on
the *released* binary: short passes (`query`, `--incremental`) get
CoreML's speed; long cold-index passes auto-fall-back to CPU and
finish without OOMing. Apple Silicon users no longer need a source
build.

## Constraints

- **No regressions for source builders.** `cargo build --release` with
  default features stays CPU-only — only the released artifact's
  feature flag changes.
- **macOS runner builds the CoreML asset successfully on GitHub-hosted
  runners.** Today the workflow uses `macos-14` (Apple Silicon). The
  build.rs already handles framework linking via `xcrun --find clang`,
  so no extra runner config should be needed — but this is the main
  thing to validate before tagging.
- **axoupdater compatibility.** The CLI's `ohara update` reads
  `dist-manifest.json`. As long as the per-target-triple asset name
  stays stable (`ohara-cli-aarch64-apple-darwin.tar.xz`), the updater
  picks up the new content transparently. Verify this against
  axoupdater 0.10.x behaviour; bump if needed.
- **Binary size.** CoreML linkage adds ~5–10 MB to the macOS-aarch64
  artifact. Acceptable. CUDA would add more (deferred regardless).

## Success criteria

A v0.6.2 release where:

- `curl … ohara-cli-installer.sh | sh` on an M-series Mac drops a
  binary whose `ohara index --embed-provider coreml fixtures/tiny/repo`
  runs to completion (proving the CoreML EP is wired in).
- The same binary's `ohara index --embed-provider auto target/perf-
  fixtures/questdb` emits the auto-downgrade `tracing::warn!` and
  proceeds on CPU — the v0.6.1 protection now actually fires for the
  released-binary audience.
- `ohara update` from a v0.6.1 install pulls v0.6.2 in place without
  manual intervention.
- Linux x86_64 / Linux aarch64 / Intel macOS artifacts are byte-stable
  vs v0.6.1 except for version bump and any unrelated commits — no
  accidental CUDA pickup, no link changes.

## Out of scope (deferred)

- CUDA-enabled Linux artifact (tracked for v0.7).
- A `--cpu-only` flag on `ohara update` that opts out of the
  accelerator default. Add when the demand surfaces.
- Smaller "thin" CoreML build that lazy-loads the framework. Not worth
  the complexity at this size.
