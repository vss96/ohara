# ohara v0.6.3 — skip-already-indexed on resume RFC

**Status:** RFC — ready to plan. Plan: `docs/superpowers/plans/2026-05-02-ohara-plan-9-resume-skip-indexed.md`.

**Goal:** stop re-embedding commits that are already in the index when
`ohara index` resumes through a merge-heavy history.

## Why this matters

The resume watermark is a single SHA stored in
`repo.last_indexed_commit`. On resume, the indexer walks
`list_commits(since=Some(watermark))` which uses
`git2::Revwalk::hide(oid)` — that only excludes the watermark and its
strict ancestor chain. Any commit reachable from HEAD via a different
path (merge from a feature branch, octopus merge, history rewrite)
gets visited again, even if it's already in `commit_record`.

Concrete case from a live QuestDB resume:

```
DB has commit_record rows:                673
Watermark + strict ancestors:             400
Commits to re-walk on resume:             273  ← wasted work
```

`commit::put` is INSERT-OR-REPLACE (v0.6.0 fix) so the work doesn't
crash — but the indexer still runs the embedder over 273 commits'
worth of hunks before discovering the row already exists. On QuestDB,
that's ~14 minutes of wasted CPU embedding per resume cycle, and it
scales with how merge-heavy the repo is.

## Shape

Add a fast pre-check in the indexer's per-commit loop: if
`commit_record.sha` already has a row for this SHA, skip the embed
and the hunk-extraction work. The watermark logic stays as-is for
abort-resume; this just adds an inner short-circuit for the
already-indexed case.

```rust
// inside Indexer::run, per-commit loop:
if storage.commit_exists(&meta.commit_sha).await? {
    continue;  // already indexed via a different parent path
}
// existing embed + hunk extraction + put
```

Storage gains one method:

```rust
async fn commit_exists(&self, sha: &str) -> Result<bool>;
```

Implemented as a single `SELECT 1 FROM commit_record WHERE sha = ?
LIMIT 1`. Uses the existing PK index — sub-millisecond.

## Constraints

- **Watermark semantics unchanged.** The check is in addition to the
  watermark, not a replacement. Initial-index runs (no watermark)
  hit `commit_exists` for every commit and get `false`, paying one
  PK lookup per commit. Acceptable: a 5,800-commit walk pays ~0.1 s
  total in PK lookups vs hours of embedding.
- **Abort-resume guarantee preserved.** Watermark still advances
  every 100 commits. Skipped commits do not advance the watermark
  on their own (we already had a row for them, so the resume
  invariant holds without an update).
- **No schema migration.** `commit_record` already has `sha` as PK.

## Non-goals

- Index-set watermark / generation-number watermark. The
  skip-if-exists check covers the same correctness space at lower
  complexity.
- Skipping hunk-only-stale rows (e.g. a commit whose `commit_record`
  exists but whose `hunk` rows were partially written before a
  crash). The existing v0.5.1 abort-resume guarantee already covers
  this — watermarks advance only after both `commit::put` and
  `hunk::put_many` succeed.
- Changing what `list_commits(since=…)` returns. We want the broader
  set; the inner skip handles the filtering.

## Success criteria

- A regression test that builds a multi-branch repo, indexes through
  some commits, kills mid-way, resumes, and asserts that
  already-indexed commits don't re-enter the embedder. Pre-fix the
  test should fail (assertion: embedder call count > 0 on the
  second run for already-indexed SHAs); post-fix it must pass.
- On the live QuestDB resume, the second-run wallclock for the first
  273 commits drops from ~14 min to under 30 s. (Spot-check, not
  a CI gate — the QuestDB perf test is `#[ignore]`'d.)
- No regression on cold-index throughput. The `quest_db_baseline`
  perf test (when re-run) shows ≤ 1% wallclock change vs v0.6.2.

## Out of scope (deferred)

- Reworking the watermark to be a proper monotonic generation number
  (would also help with shallow clones and force-pushes). Tracked
  as a v0.7 candidate.
- Adding a `--rebuild-stale` flag that explicitly re-embeds commits
  whose embedding model has changed. Different problem, different
  trigger.
