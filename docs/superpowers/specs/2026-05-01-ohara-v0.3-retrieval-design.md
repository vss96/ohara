# ohara v0.3 — Retrieval pipeline upgrade

**Status:** spec, awaiting plan.

**Goal:** raise `find_pattern` retrieval quality by replacing the v0.1–v0.2
hand-tuned linear score with the *retrieve → fuse → rerank* pipeline that
Augment, Cursor, and Sourcegraph Cody have all converged on. Add hybrid
sparse + dense retrieval and AST-aware sibling-merge chunking. Keep ohara
local-first; no new cloud dependencies.

This spec scopes only the retrieval-quality upgrades. Other ideas surfaced
during the v0.3 brainstorm (LLM-distilled commit summaries, bit-quantized
vec index, branch-reachability filter, Merkle file hashing, HyDE query
rewriting, query-understanding pre-pass) are explicitly deferred — A/B
them against the new baseline once it ships.

## Research summary

Three peers researched (one-paragraph each):

- **Augment.** Cloud-hosted personal-index for 100M+-LOC monorepos. Uses
  two-stage retrieval (bit-quantized candidate gen → float32 rerank), LLM-
  distilled commit summaries, and proof-of-possession crypto. Most relevant
  ports for us are the *retrieval-quality* ideas (rerank, hybrid).
- **Cursor.** Local AST-aware chunker (depth-first sibling merge) →
  Merkle-tree change detection → server-side embeddings on Turbopuffer.
  `@Codebase` is a multi-stage gather (semantic + grep) → cross-encoder
  rerank → reason → generate pipeline. The chunking algorithm and the
  pipeline shape are both directly portable.
- **Cody.** "Product of products" with task-specific retrieval profiles
  over a multi-retriever ensemble (Zoekt + embeddings + SCIP). A pointwise
  transformer reranker is reported as their highest-leverage stage. Cody
  Enterprise *retired* embeddings in favour of keyword + code-graph,
  citing operator complexity — supports our take that hybrid is the
  right wedge.

All three converge on three things: **multi-stage retrieve → cross-encoder
rerank**, **hybrid sparse + dense retrieval before rerank**, and **AST-
aware chunking that merges siblings up to a token budget**. v0.3 = those
three.

## Architecture

```
find_pattern(query, k=5)
    │
    ├─► vector top-100         (existing — sqlite-vec on vec_hunk)
    ├─► FTS5 hunk-text top-100 (NEW — BM25 on hunk body)
    └─► FTS5 symbol-name top-100 (NEW — BM25 on tree-sitter symbol names)
    │
    ▼
Reciprocal Rank Fusion (k=60) → top-50 candidates
    │
    ▼
Cross-encoder rerank (bge-reranker-base, opt-out via --no-rerank)
    │
    ▼
top-K → response
```

Recency stays as a *post-rerank tie-breaker* — small multiplicative
nudge applied after the cross-encoder produces its score:

```
final_score = rerank_score * (1.0 + recency_weight * recency_factor)
where recency_factor = exp(-age_days / 90.0)   // 1.0 today → ~0.37 at 90 days
      recency_weight = 0.05                    // RankingWeights default
```

Recency does not feed into RRF or the cross-encoder score directly —
relevance dominates, recency only nudges within a tight relevance band.

The existing v0.1 linear formula (`0.7·sim + 0.2·recency + 0.1·msg_sim`)
is **deleted**. RRF + cross-encoder replace it entirely; the recency
tie-breaker is the only carry-over.

## Component changes

### `ohara-storage` (largest change)

- **New FTS5 virtual tables:**
  - `fts_hunk_text(content)` indexed on hunk body text
  - `fts_symbol_name(symbol_kind, name)` indexed on tree-sitter symbol names
- **Migration V2:** populate the new FTS tables from existing `hunk` and
  `symbol` rows on first run with a v0.3 binary against a v0.2 index.
  No re-embedding required (the float32 vec_hunk table is preserved).
- **New `Storage` methods:**
  - `bm25_hunks_by_text(query, k) -> Result<Vec<HunkHit>>`
  - `bm25_hunks_by_symbol_name(query, k) -> Result<Vec<HunkHit>>`

### `ohara-embed`

- New `RerankProvider` trait:
  ```rust
  #[async_trait]
  pub trait RerankProvider: Send + Sync {
      async fn rerank(&self, query: &str, candidates: &[&str]) -> Result<Vec<f32>>;
  }
  ```
- `FastEmbedReranker` impl using `fastembed::TextRerank` with
  `bge-reranker-base` (~110 MB ONNX, CPU-only, downloads on first
  rerank call). If `fastembed-rs` does not yet export `TextRerank` for
  this model, fall back to a thin `ort` adapter.

### `ohara-core` (Retriever rewrite)

- `Retriever` becomes a pipeline:
  1. **Gather:** dispatch the three queries in parallel via `tokio::join!`.
  2. **Fuse:** RRF on the three ranking lists (free function
     `reciprocal_rank_fusion(rankings: &[Vec<HunkId>], k: u32) -> Vec<HunkId>`).
  3. **Rerank:** cross-encoder rerank top-50 (configurable). Skip if the
     `RerankProvider` is `None` (opt-out path).
  4. **Tie-break by recency:** small recency multiplier on the rerank
     score; final sort.
- The existing `rank_hits` free function is **deleted**.
- New `RankingWeights` struct loses `similarity` / `message` fields,
  keeps only `recency_weight: f32` (default 0.05) and
  `rerank_top_k: usize` (default 50).

### `ohara-parse` (chunking)

- Replace **one-chunk-per-top-level-symbol** with **AST-aware sibling
  merging up to a 500-token budget** (token = char_count / 4 approx,
  good enough for chunking decisions; not used for prompt construction).
- Algorithm: depth-first traversal of the tree-sitter parse tree.
  Accumulate sibling nodes into the current chunk while the running
  total < 500 tokens. When a single node exceeds 500 tokens, emit it
  alone (don't subdivide further — preserves semantic locality even at
  the cost of larger chunks).
- This changes the size and identity of `symbol` rows. Migration:
  on upgrade, drop and re-build the `symbol` and `vec_symbol` tables
  using the existing tree-walker plus the new chunker. Hunks are
  unaffected.

### CLI

- `ohara index --force` — explicit re-index path (used after upgrade
  when the user wants the new chunking; default upgrade only runs the
  cheap FTS-backfill migration).
- `find_pattern` MCP tool gains `--no-rerank` flag wired through to
  `Retriever`.

## Migration strategy

A v0.3 binary opening a v0.2 index sees `schema_version = 1`, runs
migration V2:
1. Add `fts_hunk_text` and `fts_symbol_name` tables.
2. Add `sibling_names TEXT NOT NULL DEFAULT '[]'` column on `symbol`
   (JSON array, populated only after `--force` re-chunk; defaulted to
   `[]` for v0.2-era rows).
3. Backfill the FTS tables from existing `hunk` and `symbol` rows
   (single-name only — sibling_names is empty until re-chunk).
4. Bump `schema_version` to 2.

Total time: ~5–10 s on a typical 50k-LOC repo (~5k hunks). No re-
embedding, no network.

The chunking change is **not** auto-applied on migration — it requires
re-walking the tree-sitter parse trees per source file (~30 s) plus
re-embedding the new chunk set (another ~10 s on CPU for ~2k merged
chunks). Gated behind `ohara index --force`. The migration log prints
a recommendation to re-run with `--force` for full v0.3 benefits, but
the index remains queryable in the meantime under the old chunk
boundaries.

## Testing strategy

- **Unit:**
  - `reciprocal_rank_fusion` — golden test against a known input/output
    matrix from a published RRF reference.
  - AST chunker — fixture file with known structure (tiny lib with 3
    functions: 50 tok, 600 tok, 200 tok). Assert chunk boundaries
    match expectation (chunk 1 = fn 1 + fn 3 merged at 250 tok;
    chunk 2 = fn 2 alone at 600 tok).
- **Integration:**
  - Extend `e2e_find_pattern` to assert the retry-pattern fixture
    *still ranks first* under the new pipeline (regression safety).
- **New e2e:**
  - Two-near-duplicate-commits fixture: same diff hash, different
    commit messages. Assert the cross-encoder picks the better-message
    one as #1.
- **Informal benchmark:**
  - End-to-end `find_pattern` latency before/after on the existing
    fixture, with and without `--no-rerank`. Record in the PR. Goal:
    P50 latency stays < 500 ms with rerank on; opt-out path stays
    < 100 ms.

## Open questions / risks

1. **`fastembed::TextRerank` availability.** Need to verify the Rust
   crate exposes `bge-reranker-base` (it does in v4.x for some models;
   confirm during plan writing). Fallback: thin `ort` adapter.
2. **Reranker download UX.** First rerank call downloads ~110 MB. The
   existing fastembed download for the embedding model already trains
   users on this; the rerank download surfaces a similar log line and
   that's it.
3. **RRF k-parameter.** Sticking with the canonical k=60 from the
   original Cormack et al. paper; revisit only if benchmark shows it
   biases too hard toward dense-or-sparse.
4. **AST chunk boundaries differ from existing symbol boundaries.**
   Symbols (Rust `fn`, Python `def`/`class`) currently map 1:1 to
   chunks; after the change, multi-symbol chunks exist. The `symbol`
   table's `name` column will hold the *primary* symbol's name, with
   merged-sibling names in a new `sibling_names: Vec<String>` field
   (serialized JSON) for FTS5 indexing.
5. **No reranker = degraded mode, not broken mode.** `--no-rerank`
   returns the post-RRF top-K directly. Document this clearly so users
   who opt out know what they're trading.

## Out of scope for v0.3 (deferred)

These came up during brainstorming. Worth doing eventually; not v0.3:

- **LLM-distilled commit summaries** (Augment) — replace `commit_msg +
  first_hunk` embedding with a "primary goal / key files / technical
  terms" summary against a local model. Speculative; depends on
  whether commit-message quality is actually our bottleneck. Defer
  until A/B against the v0.3 baseline shows it matters.
- **Bit-quantized vec index** (Augment) — premature at our scale.
- **Branch-reachability filter** (Augment) — useful for users who
  switch branches often; not many today.
- **Merkle file hashing** (Cursor) — only meaningful when v0.4 expands
  beyond commit history into current-state code.
- **HyDE query rewriting** (Cursor) — mixed evidence in benchmarks.
- **Query understanding pre-pass** (Cody) — wait for evidence that
  raw-query embeddings underperform.
- **Task-specific retrieval profiles** (Cody) — only meaningful once
  v0.3 has multiple MCP tools (currently `find_pattern` is solo).
