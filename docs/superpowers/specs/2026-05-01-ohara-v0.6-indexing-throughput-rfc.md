# ohara v0.6 — Indexing throughput RFC

**Status:** RFC — problem statement only. Design space is open; whoever
picks this up owns the architecture. The constraints and success criteria
below are the contract.

**Goal:** make a first-time `ohara index` of a real-world repo (5k–50k
commits, polyglot, hundreds of MB of pack data) finish in a tractable
time — minutes, not hours — without regressing the retrieval-quality
gains from v0.3 (three-lane retrieval + cross-encoder rerank +
AST sibling-merge chunking) or v0.4 (Java/Kotlin support).

## Evidence

Concrete data point from a v0.5.0 binary run against **QuestDB**
(Java + native, ~5_875 commits, ~830 MB main pack) on Apple Silicon
(M-series, ~10 cores), no flags:

- After ~40 min: **659 / 5_875 commits indexed (~11%)**, 11_837 hunks
  embedded, 0 HEAD symbols. Linear extrapolation: **≥6 h** for a cold
  first-time index. Probably optimistic — throughput likely degrades
  as the index grows (FTS5 insert cost, sqlite-vec index growth, WAL
  pressure), so realistic cold-start could be 8–12 h. Treat 6 h as a
  lower bound, not a target to beat by 24×.
- Index DB at that point: 131 MB main + 45 MB WAL.
- Phase shape (from `ps` sampling, not yet from real profiling):
  - First ~3 min: ~1000% CPU, ~3.3 GB RSS — parallel embedding burst.
  - After: ~100% CPU, ~100 MB RSS — single-core tail. Suggests a
    serial bottleneck in tree-sitter parsing, hunk extraction, or
    SQLite writes. Needs to be measured, not guessed.

The QuestDB run is not an outlier. Any sizeable Java/Kotlin/Spring
codebase (the Plan 4 audience) will have a similar shape. A 6-hour
cold start makes ohara hard to adopt — new users will Ctrl-C and
assume it's broken long before the index is queryable. The progress
bar from `855dc9f` makes the wait visible; it doesn't make the wait
short.

### Baseline (v0.5.1)

The implementer should treat v0.5.1 as the floor — don't re-solve
problems already solved:

- **Abort-resume safety landed in `526af35`.** Watermark advances every
  100 commits inside `Indexer::run`. `hunk::put_many` is
  DELETE-then-INSERT keyed on `commit_sha`. `put_head_symbols` was
  already replace-semantics from Plan 3 H2. The pipeline is
  end-to-end resume-safe; worst case after Ctrl-C is re-doing ≤100
  commits. Open question #5 in this RFC ("what's the resume
  contract?") is therefore answered for the linear case — only
  recency-first / pipelined paths reopen it.
- **`ProgressSink` trait** in `ohara-core` lets new phases plumb
  reporting through without coupling the core to `indicatif`.
  `NullProgress` for the MCP server, `IndicatifProgress` (TTY-gated)
  for the CLI. New v0.6 phases must hook this same trait, not roll
  their own.
- **Resource flags exist:** `--commit-batch`, `--threads` (caps
  `OMP_NUM_THREADS` + `RAYON_NUM_THREADS` before embedder load), and
  `--no-progress`. Adding more knobs is fine; rename or drop these
  with care since they're now in the public CLI surface.

## Constraints

Whatever shape the solution takes, it must hold:

1. **Retrieval quality stays at v0.3+ levels.** Three-lane retrieval,
   cross-encoder rerank, AST sibling-merge chunking, and Java/Kotlin
   annotation preservation are non-negotiable. Speed wins that come
   at the cost of measurable retrieval regressions are not acceptable.
2. **Local-first, no cloud calls.** All work happens on the user's
   machine. No cloud embedding endpoints, no remote rerank.
3. **No heavy new runtime deps.** ohara already pulls fastembed (ONNX)
   and sqlite-vec. Adding a third heavyweight runtime (CUDA, vLLM,
   etc.) is out of scope.
4. **Idempotent and resumable.** A killed index must be safe to
   restart and resume. Storage is already idempotent today — preserve
   that property end-to-end.
5. **Incremental path stays fast.** Don't slow down the
   `--incremental` short-circuit used by the post-commit hook.
6. **Single binary, no daemons.** ohara stays a CLI + stdio MCP
   server. No background services, no system-level installation.
7. **Quality gate is CI-checkable.** Constraint #1 is operationalized
   as: the existing `find_pattern_returns_retry_commit_first` e2e on
   `fixtures/tiny/repo` must pass on default settings, AND any new
   perf-flag that flips behavior (e.g. recency-first, opt-out
   reranker warmup, smaller embedder) must be A/B'd against the
   unflagged path on the same fixture, with no regression on the
   retry-pattern's rank-1 position. New perf flags introduce new
   `#[ignore]`'d e2e tests that exercise both paths.

## Success criteria

A v0.6 first-time index must hit at least **one** of these on a
reference repo of QuestDB-class size on a typical M-series laptop:

- **(A) Total time:** under **15 minutes** to fully indexed (vs. ~6 h
  today).
- **(B) Time-to-useful:** under **3 minutes** to a partial-but-honest
  index — e.g. newest N commits indexed first, older history
  backfilled — where `find_pattern` returns useful results
  immediately and `_meta` clearly exposes which window has been
  covered.

Either is acceptable. (B) is interesting because it changes the user
contract from "wait for completion" to "usable immediately, quality
improves over time"; for users who only need recent context that may
be the right product shape. Reasonable to ship both: (A) eventually,
(B) first.

## Task 0: profile (mandatory first commit)

Before architectural changes, ship a profile. Use `samply` (or
`cargo flamegraph`) against the QuestDB fixture (or another repo of
similar shape — 5k+ commits, polyglot). Output a per-phase wall-time
breakdown for: commit walk + diff/hunk extract, tree-sitter parse,
embed (ort inference), SQLite/sqlite-vec write, FTS5 insert. Paste
the numbers in a follow-up commit at `docs/perf/v0.6-baseline.md`.

This isn't optional. "It's slow" can mean any of five different
things; design decisions made without numbers will burn the budget
on the wrong knob. The profile output is the actual contract for
which design space below to invest in.

A cheap secondary measurement: **hunk-text inflation.** git2's
default diff context is 3 lines; for tiny diffs the embedder sees
6× the bytes it needs to. Count `total_diff_bytes / total_added_lines`
on the same fixture. If the ratio is high (>4), trimming context is
a free win — the embedder is the long pole and its work is linear
in token count.

## Design space (non-prescriptive)

These are directions worth considering, not a checklist or a ranking.
Pick what fits after Task 0's profile says.
- **Embedder throughput — try ONNX execution providers first.** On
  Apple Silicon, switching ort from the default CPU provider to
  CoreML is plausibly the highest-yield single knob (literature
  reports 2-3× on transformer inference; cost is ~5 lines of ort
  config + a runtime-detection branch). On Linux, CUDA / TensorRT
  if available, otherwise fall back to oneDNN / OpenVINO. This is
  the cheapest way to validate "is the embedder the bottleneck?"
  before pipeline restructuring. fastembed exposes a
  `RuntimeBuilder` knob today; ohara doesn't. Adding a
  `--embed-provider` flag (auto / cpu / coreml / cuda) is a small
  surface change with potentially large impact. Also worth: batch
  sizes, int8 quantization, ort thread-pool sizing.
- **Pipeline shape.** Today's `--commit-batch` + `--threads` exist
  but the topology (one rayon pool, many ort threads, one async
  runtime) may have contention patterns worth restructuring.
  Pipeline parallelism between walk → parse → embed → write is a
  real option; so is shrinking the serial tail by overlapping
  HEAD-symbol extraction with the commit walk.
- **Storage write path.** SQLite WAL, sqlite-vec inserts, FTS5
  inserts — any may be the serial tail. Bulk-load mode, single
  transaction per K commits, deferred index builds, vacuum-after-bulk.
- **Hunk-text inflation.** Confirm the embedder isn't being fed
  redundant text (e.g. unchanged context lines around tiny diffs);
  cheapest win may be input filtering.
- **Recency-first / partial indexing.** Change the contract instead
  of the implementation. Index newest commits first; expose the
  "indexed window" in `_meta`; allow `find_pattern` while older
  history loads.
- **Per-phase opt-outs.** Cross-encoder rerank model warmup,
  per-language parsers, etc. — anything that's free to defer until
  first query is fair game for a "fast cold start" mode.
- **External pre-embedding.** Out of scope for v0.6, but worth
  confirming the storage format doesn't foreclose a future "embed
  once, ship to teammates" workflow.

## Non-goals

- Distributed / multi-machine indexing.
- Cloud embedding or rerank fallbacks.
- Reworking the retrieval pipeline. Quality stays put; speed is the
  axis.
- GPU-required builds. CoreML / Metal acceleration on platforms that
  have them is welcome, but the binary must still run on a stock
  CPU-only Linux machine.
- Windows. Tracked separately under task-15 #2.

## Open questions for the implementer

1. **Headline metric.** Wall time on one reference repo, or
   commits/sec across a benchmark suite? Pick before building.
2. **Benchmark suite.** Lives in-repo as `tests/perf/` with
   `#[ignore]`'d wall-time assertions (opt-in via
   `--include-ignored`) so v0.7 can't silently regress. The QuestDB
   fixture is too big to vendor; recipe should be a build script that
   shallow-clones a known SHA into `target/perf-fixtures/` on demand.
   The full release CI job stays green without the perf job; a
   separate scheduled CI workflow runs the perf assertions weekly
   and posts deltas.
3. **Single bottleneck or many?** Profiling will say. If it's one
   thing, fix it. If it's a long tail, pipelining wins more than any
   single knob.
4. **Recency-first interaction with relevance.** RRF fusion already
   mixes score scales; if (B) is the path, how does the indexed-window
   boundary interact with cross-encoder rerank — penalty for
   out-of-window hits, or just a `_meta` flag?
5. **Resumability semantics.** Today the watermark is a single HEAD
   sha. If indexing becomes pipelined or recency-first, what's the
   resume contract — per-commit, per-batch, per-phase?
