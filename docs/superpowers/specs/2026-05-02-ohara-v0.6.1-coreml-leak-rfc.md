# ohara v0.6.1 — CoreML memory leak RFC

**Status:** RFC — investigation. Scope is "find and fix the leak", not
"ship a new feature". Whoever picks this up owns the diagnosis.

**Goal:** make `--embed-provider coreml` viable for cold first-time
index runs on Apple Silicon. Today it OOM-kills (jetsam) at ~32 GB
compressed memory on a QuestDB-class repo (~5,800 commits) within
the first ~minutes — `--embed-provider cpu` runs the same workload
with stable RSS in the low hundreds of MB.

## Evidence

From the v0.6.0 baseline attempt on QuestDB:

```
2026-05-02 01:58:21 kernel: [com.apple.xnu:memorystatus]
  memorystatus: killing largest compressed process ohara [55727] 32229 MB
```

Surrounding context the kernel logged before the kill:

- **Hundreds of `ANE_PowerOff_gated_block_invoke` / `ANE_PowerOn_gated_block_invoke`
  cycles per second** under `clientname:ohara` — the Apple Neural
  Engine is being toggled per inference batch.
- Periodic `ANE_SaveState_block_invoke: WARN:: clientname:ohara is
  keeping programs loaded and not freeing - isPowered: 1 Programs
  Open: 1 Wired-memory: 65536 B`.
- ort logged at startup: `CoreMLExecutionProvider::GetCapability,
  number of partitions supported by CoreML: 97 number of nodes in
  the graph: 623 number of nodes supported by CoreML: 447`. 97
  partitions = many CPU↔ANE handoffs per inference.

Crucially, `--embed-provider cpu` on the same fixture and the same
binary did **not** leak. Both the v0.5.1 abort-resume work and the
v0.6.0 commit::put fix landed before this run — so the leak is
specifically in the CoreML execution path, not in the indexer or
storage layer.

The same binary handles short-lived `ohara query` and `ohara index
--incremental` calls fine (a few inference batches at most). The
leak emerges over **thousands of small-batch inferences** in a single
process.

## Hypotheses (in rough priority order)

1. **ort caches CoreML graph state per inference batch.** The 97-
   partition graph is replanned / re-cached on each call, and the
   per-batch state is never freed because the underlying ANE
   "Program" stays open (`Programs Open: 1`).
2. **fastembed's `TextEmbedding::embed` retains the per-call
   intermediate tensors** when an execution provider was attached,
   even though the CPU-only path drops them.
3. **CoreML.framework leaks ML model compilations on macOS 26.x**
   under repeated `MLModel.compileModel` calls. The ANE
   `SaveState` warning about "keeping programs loaded and not
   freeing" hints at this.
4. **A small fastembed/ort interaction** — e.g. an `Arc<Session>`
   that keeps growing because each call adds a new IO binding to a
   collection on the session.
5. **Our own code holds embedder output longer than expected** —
   unlikely (CPU runs are fine with the same Indexer loop), but
   worth confirming.

## Constraints

- **No regression on CPU or CUDA paths.** Those run today; whatever
  fix lands must not change their resource shape.
- **No new heavyweight runtime deps.** If the fix is "swap CoreML
  for a different ML runtime", that's out of scope; this RFC is
  scoped to making CoreML usable as it is.
- **Quality gate stays in place.** The paired e2e test from Plan 6
  (`tests/perf/embed_provider_paired.rs`) must continue to pass —
  CoreML and CPU runs return the same rank-1 hit on the retry
  fixture.
- **Ship-able even if upstream fix is needed.** If the root cause is
  in fastembed or ort and a same-week fix isn't viable, the v0.6.1
  release is a documented workaround + an auto-detect change that
  prefers CPU for long index passes on Apple Silicon (and CoreML
  for short-lived query / `--incremental` calls).

## Non-goals

- Fixing CUDA leaks (no NVIDIA box on hand to reproduce).
- Reworking the embedder pipeline (Plan 6 Phase 2B Task 5).
- Bumping fastembed or ort to newer versions just to get a fix —
  acceptable only if the upgrade is the fix and the upgrade is
  self-contained.

## Success criteria

A v0.6.1 cold index of QuestDB with `--embed-provider coreml` must
either:

- **(A)** Finish without OOM, peak RSS staying below 4 GB for the
  duration of the index pass; **or**
- **(B)** If the root cause is genuinely upstream and unfixable in
  this release window, ship a documented workaround (probably:
  `--embed-provider auto` resolves to CPU when an `index` pass
  would walk more than `N` commits — same threshold the
  `--incremental` short-circuit uses). The flag still works for
  `query` and short `index --incremental` calls.

(B) is the explicit fallback so v0.6.1 has a non-empty deliverable
even if the diagnosis takes more than a week.

## Investigation framework

Phase 1: **diagnose.** Reproduce the leak in the smallest possible
harness, identify which allocations grow without bound, name the
hypothesis from §"Hypotheses" that matches the data.

Phase 2: **fix or work around.** Either land the upstream-style fix
or implement (B).

The plan file (`docs/superpowers/plans/2026-05-02-ohara-plan-6.1-coreml-leak.md`)
spells out concrete probing commands.

## Out of scope (deferred)

- Symbol-lineage support in `explain_change` (originally planned
  alongside v0.5; revisit when the throughput story is solved).
- Profile-driven autotuning of `--commit-batch` / `--threads` based
  on observed RAM pressure.
